# Arizuko: minimum local deployment for a working chat agent

Repo: `/home/onvos/app/arizuko` (fresh clone, read-only research).

## 1. Minimum viable daemon set

Production compose is generated by `compose/compose.go:Generate` (`compose/compose.go:376`).
Three daemons are **always** emitted, unconditionally (`compose/compose.go:558-564`):

- **authd** — auth authority. Owns `auth.db` only. Serves JWKS, mints ES256
  service/session tokens, hosts `/auth/*` OAuth login *only if* `AUTH_BASE_URL`
  is set (`authd/oauth.go:39-42`: `registerOAuth` returns immediately if
  `cfg.AuthBaseURL == ""`). No docker socket, no message DB
  (`compose/compose.go:718-721`).
- **routd** — the canonical router / conversation state. Owns `routd.db`,
  exposes `/v1/send`, `/v1/channels` etc., verifies callers via authd's JWKS,
  and dispatches turns to runed. No crackbox, no docker socket
  (`compose/compose.go:741-745`). This is what a CLI/adapter/webd all talk to.
- **runed** — the execution plane. **The only daemon wired to the Docker
  socket** (`/var/run/docker.sock` bind-mount, `group_add` into the docker
  gid) plus crackbox + per-folder agent networks (`compose/compose.go:768-799`).
  Spawns one container per agent turn.

Everything else is conditional:
- **webd + proxyd + vited** — only emitted if `WEB_PORT` is set in `.env` and
  `PROFILE != minimal` (`compose/compose.go:565-570`). This is the browser
  chat surface (`/chat/`, `/panel/`, `/api/`, `/mcp` routed through proxyd to
  webd, `compose/compose.go:301-309`).
- **timed, dashd, davd, onbod** — only when `PROFILE` is `standard`/`full`
  (`compose/compose.go:571-582`); `PROFILE=web` or `PROFILE=minimal` skip
  all of them.
- **crackbox** (egress sandbox for the agent) — only if `CRACKBOX_ADMIN_API`
  is set (`compose/compose.go:586-588`); otherwise `EgressConfig.active()`
  is inert and the agent container runs on the default bridge network
  (confirmed by comment in `runed/docker.go:123-129`).
- **Channel adapters** (teled/discd/slakd/mastd/bskyd/emaid/whapd) — each
  needs its own `template/services/<adapter>.toml` copied into the instance's
  `services/` dir (`INSTALL.md:177-192`); none is required for a chat pane
  since the CLI (`arizuko send`, see §5) and webd both bypass adapters
  entirely.

**Bottom line for "a chat agent I can send a message to and get a reply
from, locally":** `authd + routd + runed` is sufficient — no adapter, no
webd, no OAuth, no crackbox. Use the CLI `arizuko send ... --wait` path
(§5). If you also want a browser chat pane, add `webd + proxyd + vited`
(set `WEB_PORT`), still no adapter/OAuth required for the operator-direct
CLI path, but webd's own login UI (`webd/server.go:66` redirects
unauthenticated browser hits to `/auth/login`) does need a real OAuth
provider configured in authd (Google/GitHub/Discord client id/secret,
`authd/oauth.go:56-` onward) since there is no dev/bypass login — grep of
`authd/oauth.go` and `core/config.go` found no `DEV`/insecure/bypass login
path.

## 2. Bootstrap commands, env vars, state layout

```bash
git clone https://github.com/kronael/arizuko && cd arizuko
make build                 # ./arizuko CLI + all daemon binaries, host-side
sudo make images           # builds arizuko:latest (all daemons) + arizuko-ant:latest (agent)
                            # + adapter/vite/davd/ttsd/crackbox images (extra, harmless if unused)
./arizuko create mybot     # creates /srv/data/arizuko_mybot/ (or $PREFIX/arizuko_mybot/)
```

`arizuko create` (`cmd/arizuko/main.go:212-` `cmdCreate`) auto-generates
`.env` with `AUTH_SECRET` and `SECRETS_KEY` (32-byte random hex each,
`cmd/arizuko/main.go:257-270`) — no manual secret generation needed. State
directory: `/srv/data/arizuko_<name>/` (override with `PREFIX` env var,
`INSTALL.md:90-95`), containing `.env`, `store/` (sqlite DBs), `services/`
(adapter TOMLs), `groups/`, `ipc/`.

Required `.env` additions (`INSTALL.md:133-159`):

```bash
ASSISTANT_NAME=MyBot
ANTHROPIC_API_KEY=sk-ant-...          # or CLAUDE_CODE_OAUTH_TOKEN=...
```

For a browser chat pane, also add `WEB_PORT=8095` (or similar) to enable
webd+proxyd+vited (`compose/compose.go:565-570`); if you want the minimal
daemon set (no timed/dashd/davd/onbod) also add `PROFILE=web` — otherwise
`PROFILE` defaults to `full` (`compose/compose.go:552`) and those extra
daemons come up too.

Group/folder setup (a "folder" = one agent instance) happens automatically:
`create` seeds a default group `main` (`cmd/arizuko/main.go:283`
`s.PutGroup(core.Group{Folder: "main", ...})`), so `arizuko send mybot main
"hi"` works immediately with zero adapter config.

Run:

```bash
./arizuko run mybot        # generates docker-compose.yml, `docker compose up` (foreground)
```

`docker` socket access is required by the operator (`INSTALL.md:67-79`):
add user to the `docker` group, or run arizuko/docker commands with `sudo`.

## 3. Docker-per-turn — is it mandatory?

**Yes, mandatory in production. There is no non-Docker path to run the
Claude Code agent.**

- `runed/cmd/runed/main.go:97`: `runtime := runed.NewDockerRuntime(cfg,
  folders)` — this is unconditional; there is no flag or env var that
  switches it.
- `runed/docker.go:22-49`: `dockerRuntime` is documented as *"the
  production Runtime: a pure container-spawner"*, and `NewDockerRuntime`
  wires `runner: container.DockerRunner{}` plus `exec.Command(container.Bin,
  ...)` calls for signal/kill (`runed/docker.go:44-56`) — real `docker`
  CLI invocations, no in-process exec fallback.
- The only non-Docker implementation is `FakeRuntime`
  (`runed/runtimes.go:13-30`), explicitly documented as *"backs unit tests
  of the run envelope without spawning anything... Never used in
  production"* (also stated of the companion `staticBroker`,
  `runed/runtime.go:86-91`). It is wired only from test code, never from
  `runed/cmd/runed/main.go`.
- `Runtime` interface doc comment confirms the split explicitly
  (`runed/runtime.go:60-65`): *"production wraps container.DockerRunner +
  the ipc MCP host; LocalRuntime/FakeRuntime back CI + unit tests without
  docker."*

So: for every agent turn, `runed` shells out to `docker run` to spawn a
fresh, one-shot `arizuko-ant:latest` container (`compose/compose.go:772-799`
mounts `/var/run/docker.sock` into runed for exactly this). There is no
"local exec" or "in-process Claude Code SDK call" mode — Docker is load
bearing for every reply.

### The container image (`ant/` → `arizuko-ant:latest`, built by `make agent` / `Makefile:122-123`)

`ant/Dockerfile` (`node:22-slim` base, ~200 lines): installs Chromium +
Playwright deps, a large general toolbelt (ripgrep, jq, ffmpeg, ImageMagick,
pandoc, tesseract, sqlite3, postgres/redis clients, Go 1.24, Rust via
rustup, uv+Python 3.14, gh CLI, yt-dlp, and more — `ant/Dockerfile:9-84`),
then `npm install -g ... @anthropic-ai/claude-code @apify/mcpc
@openai/codex` (`ant/Dockerfile:171-173`) and
`npm install @anthropic-ai/claude-agent-sdk@latest` (`ant/Dockerfile:188`,
explicitly *not* pinned — "auto-updates to latest on every image build").
Runs as non-root `node` uid 1000 (`ant/Dockerfile:206-207`).
`ENTRYPOINT ["/app/entrypoint.sh"]` (`ant/Dockerfile:212`); `ant/entrypoint.sh`
compiles TypeScript on the fly, reads secrets/prompt as **stdin JSON**, runs
`node /tmp/dist/index.js`, and the container is one-shot: *"stdin JSON in,
stdout JSON out"* (`ant/Dockerfile:209-211`).

Requirements to spawn it:
- **docker.sock**: bind-mounted only into `runed` (`compose/compose.go:784
  '/var/run/docker.sock:/var/run/docker.sock'`); `runed` is put in the
  docker group's gid (`compose/compose.go:781 dockerGID()`) so it can spawn
  containers as uid 1000.
- **crackbox egress**: optional. `runed/docker.go:123-140` builds an
  `EgressConfig` from `runed`'s own config every spawn; if
  `d.cfg.EgressAPI` (i.e. `CRACKBOX_ADMIN_API`) is empty the config is inert
  and the agent container runs on the default network with normal internet
  egress — crackbox is a sandboxing add-on, not a hard requirement for a
  working reply.
- **per-folder agent network**: `runed` creates one per group at spawn time
  (`container/network.go`, not required reading for this report — noted for
  completeness).

## 4. What one turn costs to run locally

- **Needs `ANTHROPIC_API_KEY` or `CLAUDE_CODE_OAUTH_TOKEN`.** Both are in
  the fixed secrets allowlist (`store/secrets.go:104-105`) and are
  documented as *"operator anchors... required for the Claude Code SDK to
  call the LLM"* (`container/runner.go:252-253`). They are injected as
  container env at spawn (`container/runner.go:710` lists them alongside
  the folder-secret plumbing). A web-chat user can BYOA (bring your own
  `ANTHROPIC_API_KEY`) via `dashd`'s env-profile UI
  (`dashd/me_env.go:19-20`, `dashd/profile.go:68`) which shadows the
  folder/operator default for that user's spawns
  (`routd/dispatch.go:439`) — irrelevant for the minimal local case, just
  set the operator-level key in `.env`.
- **Claude Code binary**: baked into the agent image at build time
  (`@anthropic-ai/claude-code` global npm install, `ant/Dockerfile:171`) —
  not something you configure per instance; it ships with whatever image
  `arizuko-ant:latest` was last built from (rebuild the image to pick up
  newer Claude Code releases).
- **Cost per turn**: whatever the Claude Code SDK bills against
  `ANTHROPIC_API_KEY` for that turn's tokens/tool-calls — no arizuko-side
  metering was inspected here beyond the cost cap comment at
  `core/config.go:115` (`store/cost_log`, "set false to bypass" a budget
  gate) — out of scope for this report; flagging as worth a follow-up read
  of `cmd/arizuko/budget.go` if cost caps matter for the integration.

## 5. Smallest "hello world" — commands, end to end

This is the fastest path to a working reply, with **no adapter, no webd,
no OAuth, no browser** — just `authd + routd + runed` and the CLI.

```bash
# 1. build
git clone https://github.com/kronael/arizuko && cd arizuko
make build
sudo make images            # or the minimal subset: arizuko + arizuko-ant

# 2. create instance (auto-generates .env with AUTH_SECRET/SECRETS_KEY,
#    seeds a default group "main")
./arizuko create mybot

# 3. add the one required key
#    edit /srv/data/arizuko_mybot/.env:
#      ASSISTANT_NAME=MyBot
#      ANTHROPIC_API_KEY=sk-ant-...
#    (optionally PROFILE=web to skip timed/dashd/davd/onbod if WEB_PORT
#    is later added — irrelevant here since no WEB_PORT is set, so webd/
#    proxyd/vited are never emitted regardless, per compose.go:565-570)

# 4. run (foreground; brings up authd + routd + runed only, since no
#    WEB_PORT/adapter services/ toml is configured)
./arizuko run mybot &

# 5. send a message and wait for the reply — writes straight into routd.db
#    on web:main (operator DB authority, no token/adapter needed), routd's
#    poll loop feeds runed, runed docker-spawns arizuko-ant, agent replies,
#    CLI polls BotRepliesSince and prints the reply until quiescence
#    (cmd/arizuko/send.go: operatorInject + pollRound/streamRound)
./arizuko send mybot main "hello" --wait
```

`cmdSend` (`cmd/arizuko/send.go:19-30` doc comment) documents exactly two
paths: *"operator-direct (default): no chat token → inject straight into
the DB on web:<folder>... the gateway poll loop runs the agent"* vs.
*"chat-token: ... POST to the public /chat/<token> endpoint via WEB_HOST"*
(the browser/webhook path, needs `WEB_HOST` configured — `cmd/arizuko/
send.go:110-112` dies if unset). The operator-direct path
(`operatorInject`, `cmd/arizuko/send.go:161-` ) is what `--wait` follows:
it polls for the agent's replies for up to 120s (`idleStop`) then 15s of
quiescence (`quietStop`) and prints them (`cmd/arizuko/send.go:189-193`).

There is also `./arizuko chat mybot` (`cmd/arizuko/main.go:881-921`) — an
interactive REPL that shells out to the **host's own** `claude` CLI
wired via `socat` to the instance's IPC unix socket
(`ipc/main/gated.sock`) as an MCP server. This requires `claude` and
`socat` in the host `$PATH` (`cmd/arizuko/main.go:889-894`) and is a
separate, host-side Claude Code session talking to arizuko's IPC/tool
surface — not the same thing as a routd/runed-dispatched turn, and not
needed for the minimal "send a message, get a reply" loop above.

## Caveats / things not fully verified

- Did not trace routd's "gateway poll loop" implementation (mentioned in
  `cmd/arizuko/send.go` comments) to confirm its poll interval — first
  reply latency after `arizuko run` may include normal poll-cycle lag plus
  cold container start (the CLI's own 120s `idleStop` budget suggests
  cold-start can be nontrivial).
- Did not verify whether `PROFILE=web`/`minimal` changes are honored by
  `arizuko run`'s generated `docker-compose.yml` on every re-run vs. only
  at first `create` — `compose.Generate` is read from `.env` each time
  (`compose/compose.go:552 envOr(env, "PROFILE", "full")`), so editing
  `.env` before `arizuko run` should be sufficient, but this wasn't traced
  through `cmd/arizuko/main.go:cmdRun`.
- Cost-cap / budget gating (`cmd/arizuko/budget.go`, `core/config.go:115`)
  not read in depth — worth checking before wiring unattended traffic from
  the trading terminal's chat pane.
