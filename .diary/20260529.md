## 06:50

Systematic UDP-loss investigation (user: "udp has to work without packet
loss"). Proved with kernel counters: RcvbufErrors/InErrors/SndbufErrors +0,
OutDatagrams==InDatagrams under load. Loopback UDP is lossless — "packet
loss" was a misdiagnosis. Two distinct diseases below.

## 06:52

Disease 1 (FIXED): rsx-mark busy-spun 97% CPU but was unpinned. CFS floated
it onto gateway's pinned core 1 → gateway starved (0.4%) → couldn't drain UDP
socket → 16MB backlog → kernel RcvbufErrors (real drops). Fix: mark ergonomic
(sleep 250µs, 97%→4%), unpinned like other off-path svcs. Documented 6-core
layout in README + CLAUDE.md. Drops → 0. Also fixed marketdata FAULTED-panic
crash loop (RSX_MD_REPLAY_ADDR unset, stream_id=1 vs ME stream 10) and
WAL-replay retry on NOT_AVAILABLE (10ms flush window).

## 06:55

Disease 2 (DIAGNOSED, queued bugs.md SEQ-1): residual ~30 FAULT/min with zero
packet loss = casting seq-space collision. ME stamps BBO from CastSender's own
next_seq (send()) but WAL'd records from WAL seq (send_framed); BBO not WAL'd
→ counters desync → wire seq regresses → "sender reset detected"/FAULTED. Plus
filtered-fan-out holes: OrderDone cmp-only, OrderFailed neither, risk forwards
subset to gw, risk margin-reject sent with seq=0 (gateway drops seq==0). Fix
recommended: full fan-out on single WAL seq. Needs sign-off — touches ME/risk/
gateway/WAL/replay.

## 07:30

SEQ-1 FIXED + verified. Three-part fix: (1) ME publish_events fans OrderDone/
OrderFailed/BBO to both streams on the WAL seq (BBO was on CastSender's own
counter — the desync root cause). (2) ME main loop fans OrderAccepted (was
WAL-only) + duplicate-reject (was cmp-only) to both — OrderAccepted per resting
order was the ~2/sec residual. (3) risk forward_to_gw renumbers the gateway
stream with gw_sender's own contiguous seq (fixes seq=0 margin-reject + ME-seq
holes; gateway never replays from risk so safe). Result: FAULTs 0/15s on all
streams; e2e probe 5→551 samples, 0 fail. Exchange works end-to-end.
