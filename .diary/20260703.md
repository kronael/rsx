## Q&A this session

- **QUIC lifecycle** (why BENCH-QUINN-ACCEPT-BI panics): QUIC nests a
  connection lifecycle (TLS-1.3 handshake in-transport → Connection → close/
  idle-timeout) and a stream lifecycle. `open_bi()` returns the stream pair
  LOCALLY and sends nothing on the wire; the peer's `accept_bi()` only resolves
  once the opener WRITES (first STREAM frame). The bench accepts before any
  write → unwrap panics. Fix: client writes a priming byte before server
  accept.
- **TUI is Rust/ratatui, not Go.** Deciding factor: reuse rsx-types/rsx-messages
  wire types directly (one source of truth) vs a Go/Bubbletea client that
  re-declares every wire struct + webproto framing. Also one workspace/toolchain
  and ratatui's TestBackend gives deterministic UI tests.

## latency panel — net/internal/engine timing breakdown

Founder: the TUI should display timings to shill the exchange's speed, and every
op should measure network + internal-network + engine time. Added
`GwEvent::Latency{net_ns,internal_ns,engine_ns}`:
- **net** = client↔gateway (QUIC leg), measured by the client;
- **internal** = casting GW→Risk→ME→Risk→GW (stamped by the gateway);
- **engine** = ME match + risk (stamped by the gateway).
App folds it into a rolling window (last + p50 + best); render shows a prominent
speed strip (`⚡ RTT 10.4 µs = net 2.5 µs + internal 7.6 µs + engine 340 ns · p50
… · best …`), adaptive ns/µs/ms formatting with integer math (no floats).
QuicConn measures the round-trip and fills the net leg from the server-stamped
internal+engine (FIFO-paired via a pending-Instant queue). Loopback test
extended: server emits a Latency frame, client asserts it measured net>0 with
internal/engine intact. Demo seeds honest measured figures (match ~340ns,
casting RTT ~7.6µs). 18 rsx-tui tests green, clippy clean.
