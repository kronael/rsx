**Two analyses (read-only, get-back-to-me):**
- gateway↔marketdata symmetry (opus): two forks of one monoio shape; filed
  alignment findings to BUGS.md (MD-EGRESS-STALL / GW-OUTBOUND-UNBOUNDED are
  mirror halves; WS-fork-drift; env-doc drift).
- monoio-vs-tiles (fable): **keep tokio off-path** (replication disk-bound, PG
  drain-bound, mark tolerant); **don't tile gateway/marketdata.** Killer
  finding: the ms-scale WS latency is not the reactor — monoio 0.2.4's timer
  wheel is ms-granular and rounds up, so the gateway's "500µs" egress poll
  fires at ~1–1.5 ms; marketdata has no egress wake at all. Fix = notify-based
  egress wake (0 extra cores). A full tile split saves ~5–15µs p50 for 4 cores
  the box lacks; loop-arch-bench already measured busy-spin tiles worst
  (syscall-bound; batching, not pinning, drives throughput). To file:
  MD-EGRESS-NO-WAKE + the monoio-timer root cause.

**Lesson (the hard way):** my rapid doc commits raced the tui sub's shared git
index — `git add X && git commit` swept the sub's staged deletions into a docs
commit (77e2961). Nothing broke (all wanted changes) but the history is
mislabeled and unfixable without amend/rebase (forbidden). Rule: **never
`git commit` while a code sub commits on the same tree**; if unavoidable, use
an explicit-pathspec commit (`git commit <file>`, only-mode) so it can't sweep
the index. Holding BUGS.md (fable + codex findings) + the publish roadmap
(book → matching → gateway+marketdata → tui → hosting/install-run → blog)
until the tui-fix sub finishes.
