### Landed the two parked cast recovery benches

**Outage recovery** (`outage_recovery.rs`): big-gap recovery over the TCP
replication cold path. The parked WIP hung for two reasons, both harness
bugs: (1) the `run_once` callback never returned `false` on
`RECORD_CAUGHT_UP`, so it tailed the live replication feed forever (the
server tails live after catch-up by design); (2) the dark window was a
wall-clock firehose (~100k rec/s) → millions of records → the server's
per-connection WAL rescan never finished. Fix: stop on CAUGHT_UP + pace
the gap to `outage_secs × RATE`. Result: 7.5k-record gap ~260–790ms
(drifts up across cycles as the active WAL grows — the REPLAY-RESCAN cost,
now confirmed), 52k-record gap ~1s. Commit 25c9aa2.

**Loss tolerance** (`loss_degradation.rs`): how much sustained loss the
live NAK path survives. Key insight — the ceiling is set by the retry
COUNT (`max_nak_retries`), since a gap dies only after that many
*consecutive* retransmit losses (~loss^retries), so retransmits must be
allowed to drop too. Two traps found and fixed: (a) 144B fills bypass the
send ring → O(N) WAL-scan retransmits, so a short debounce out-runs its
own retransmit and faults spuriously → switched to 64B ring-served
records; (b) even then, debounce < retransmit-round-trip-under-jitter
caused non-monotonic spurious faults (1% worse than 5%) → bumped debounce
to 5ms (jitter-immune; debounce sets speed, not the ceiling). Result
(default 8-retry budget, N=10k): reliable through **30% loss** (5/5,
throughput degrading ~75×), collapse at 40%. Far beyond the "≤0.01%"
naive figure. Commit 50c78f9. Report:
`reports/20260707_cast-loss-recovery.md`.

## 17:47 2026-07-07 — rsx-book correctness audit (fable) + reachability

Ran a fable sub to hunt bugs in `rsx-book`. Found **4 confirmed** correctness
bugs (runnable repro compiles against real rsx-book; the ME-crash one panics

maker side), the per-side occupancy bits, and the BBA px cache all assume one
side / one price per level. Existing smooshed-tick tests only used same-side
bands, so the class was invisible.

- BOOK-MIXED-SIDE-SELF-TRADE (CRIT): SELL taker fills a resting SELL → position
  corruption. BOOK-STALE-OCC-ME-CRASH (CRIT): stale occupancy bit → `emit_bbo`
  derefs empty head → ME panic (`slab.rs:92`) → WAL-replay crash loop
  (live-verified). BOOK-FOK-CLAMP-DIVERGENCE (HIGH): zone-0 price aliasing →
  FOK feasibility passes, under-fills → Fill+OrderFailed for one order in
  release. BOOK-STALE-BBA-WRONGFUL-POSTONLY (MED).

**Reachability reframing (founder's question — "doesn't it recenter first?").**
"Side" in the compression map = above/below **mid**, not buy/sell; a buy and a
sell can both sit below mid and, in a coarsened band (zone ≥1, ≥5% from mid),
share a slot. Zone 0 (±5%) is 1:1 (`compressions=[1,10,100,1000,1]`), so mixing
CAN'T happen at the touch **while the touch is near mid**. The engine is correct
as long as the touch stays in the 1:1 zone — an implicit, unasserted invariant.
And its guarantor is **unwired**: `trigger_recenter` has no production caller
(bench-only), `should_recenter` is computed but never acted on, `migrate_batch`
only drains a migration nobody starts. So the seed mid is pinned forever; a
stale seed or a >5% move parks the touch in a coarsened zone and the bugs become
reachable (the repro: mid=50k, trading at 47k ≈ zone 1). Fix directions offered:
(1) wire recentering + assert touch-is-1:1 (design-preserving, cheap hot path,
but the migration path itself has bugs — fable #5); (2) make match/occupancy/BBA
side+price-aware (robust, complicates hot path); (3) constrain resting to zone 0.
Awaiting founder call before any fix.
