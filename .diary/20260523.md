# 2026-05-23 — Measurement sprint: hangs, harnesses, send-path

## TL;DR

The component landscape was untrustworthy — three reported
numbers contradicted each other (`udp_rtt 29µs`,
`cmp_one_way: harness hung`, `in-process e2e 9µs`).
Spent the day reconciling. Result: all four benches now
tell the same story, and the headline matching round-trip
floor is **10.1 µs in-process** with the send path
(CmpSender::send body) eating 78% of that.

## What was wrong

### Bench harness #1: `udp_rtt_bench` overstating by 4×

`echoer` thread called `set_read_timeout(Some(50ms))` on
every iteration to make the blocking `recv_from` pollable
against a stop-channel. That added a setsockopt syscall
per iter (~µs) plus blocking-recv wake-up overhead. Result:
29 µs measured, 7.3 µs real.

Fix: non-blocking sockets, busy-spin, sentinel byte for
shutdown. Same harness shape as the in-process e2e bench
which DIDN'T have the overstatement. Cf.
`rsx-dxs/benches/udp_rtt_bench.rs`.

### Bench harness #2: `cmp_one_way_bench` + `cmp_rtt_bench` hung

CMP has a peer-consumption-driven flow-control window
(`peer_consumption_seq + peer_window`, default 64 K).
Without `sender.tick()` + `receiver.tick()` calls, the
sender never receives the peer's status messages that
advance `peer_consumption_seq`, so its window closes
around iteration 65 537 and `send()` returns `Ok(false)`
forever.

Fix: drive `tick()` + `recv_control()` once per 1024
sends. Same trick the in-process bench was already using.
Reconciled numbers: cmp_one_way = 3.95 µs, cmp_rtt = 10.3 µs.

### Bench harness #3: the e2e bench was a 614-line monolith

Renamed `bench-e2e-pipeline` → `bench-match-rt` (it's
not e2e — risk, gateway WS, marketdata are all out of
scope). Rewrote at ~450 lines with **per-stage timing**:
nine timestamps per iteration (gw_send_start through
gw_recv), 8 deltas measured, p50/p95/p99/max per stage.
Same in-process pattern.


**The CMP send path is the dominant cost.** 7.8 µs out of
10.1 µs total round-trip is just two sendto syscalls + CRC32
+ NAK ring-cache copy. The ME algorithm itself (dedup +
match + WAL) is **320 ns total** — three orders of magnitude
below the 1 128 µs production p50.

Production p50 = 1 128 µs / floor = 10 µs → **~110× overhead
budget** is sitting in CMP plumbing + process boundaries +
reactor scheduling.

## Send path analysis

`CmpSender::send` body (rsx-dxs/src/cmp.rs:185-238) does:

1. Flow-control check (free)
2. `set_seq` + `as_bytes` (free)
3. `compute_crc32(payload)` over 128 bytes via `crc32fast`
   (SIMD-accelerated; ~30 ns)
4. `WalHeader::new` + `to_bytes` (~20 ns)
5. Two `copy_from_slice` (header → buf, payload → buf;
   ~10 ns each)
6. **`socket.send_to(&buf, dest)` — sendto syscall.** Dominant
   cost; loopback UDP sendto is ~1.5-2 µs.
7. NAK ring-cache `copy_from_slice` (144 bytes; ~10 ns)

Total: ~30 + 20 + 20 + ~1500-2000 + 10 = **~1.5-2.1 µs of
real work, mostly in the kernel**. The bench shows 3.9 µs
which adds ~2 µs of measurement / Rust-side overhead per
call (could be timer reads, branch mispredicts, cache
misses).

## Implications

- **Production gateway uses monoio (io_uring).** Its `send_to`
  goes through the io_uring submission ring instead of the
  classical sendto syscall — should be ~half the cost
  (kernel-bypass-adjacent). The bench's `std::net::UdpSocket`
  pays the full syscall overhead. Production gateway side
  is likely closer to 2 µs per send vs the bench's 3.9 µs.
- **CRC32 is NOT the bottleneck.** `crc32fast`'s SSE4.2
  PCLMUL implementation is fast.
- **The send-ring copy is also negligible.** 144-byte
  copies are sub-100 ns.
- **The sendto syscall itself is the structural floor.**
  Eliminating it requires either kernel-bypass (DPDK,
  AF_XDP) or batching (sendmmsg, io_uring SQE-batched). The
  project's "Later: DPDK/AF_XDP swaps I/O layer" path is
  what gets us below this floor.

## Sleep audit findings

`.ship/19-SLEEP-AUDIT/SLEEPS.md` from the sub today:
**57 sleep/timeout sites catalogued; only 2 are bugs**.

Both are the known sleep-as-yield pattern in monoio CMP
poll loops:
- `rsx-gateway/src/main.rs:407` — costs ~655 µs at p50
- `rsx-marketdata/src/main.rs:328` — same pattern, lower
  priority (off the order critical path)

The audit confirmed by oracle (codex) that monoio's timing
wheel has 1 ms slot resolution, so a `sleep(100µs)` rounds
up. Tactical fix: `monoio::task::yield_now().await` (the
`yield_now` I wrote earlier in `rsx-types` starved the
accept task — that approach was wrong; monoio's own
`yield_now` would work, but check monoio 0.2.4 first).
Strategic fix: monoio-native UdpSocket + readable() on
CmpReceiver (~50 LOC).

**Everything else is fine.** Persist-worker tokio sleeps
in `rsx-risk/src/persist.rs` are on a dedicated
`current_thread` runtime, off the shard's order path —
confirmed by reading `rsx-risk/src/main.rs:318` (which
spawns it via `std::thread::spawn` not as a tokio task).

## What I learned today (the wisdom)

1. **Bench reconciliation is the actual measurement.** If
   three benches contradict each other, none of them is
   measuring what you think. The disagreement IS the
   signal.
2. **Setsockopt and blocking-recv wake-ups cost real µs in
   tight loops.** Per-iter `set_read_timeout` is invisible
   in code review and overstated UDP RTT by 4× for weeks.
3. **CMP flow control needs both peers to tick().** Not
   obvious from the API; the bench harnesses learned the
   hard way.
4. **The matching algorithm is fast. Everything around it is
   slow.** dedup + match + WAL = 320 ns. Send + recv +
   syscall = 8 µs. Production-vs-floor = 110×.
5. **`yield_now` in monoio is dangerous.** A naive
   `cx.waker().wake_by_ref(); Poll::Pending` puts the task
   at the back of the ready queue, but if there are no
   other ready tasks and the reactor never re-polls
   io_uring, the listener task starves. Reverted in
   `9bbb8f6`.
