## TUI access — SSH forced-command dispatch + web-terminal spec

Task: make rsx-tui reachable over SSH and the web so any registered key
gets its own live session against rsx.krons.cx.

**SSH path — implemented (git-shell/gitolite pattern).** One shared
restricted unix user `rsx-tui`; its authorized_keys is the key→user
table. Each line: `restrict,pty,command="/usr/local/bin/rsx-tui-dispatch
<user_id>" <pubkey> <comment>`. `restrict` = deny-all (no port/agent/X11
fwd, no user-rc, no pty), `pty` re-adds the one thing ratatui needs — the
only deviation from git-shell's `no-pty`. `command=` forces the launcher
and drops SSH_ORIGINAL_COMMAND. user_id baked in the line = single source
of truth, no second table to drift.
- `scripts/rsx-tui-dispatch`: validates user_id is bare int, sources
  `/etc/rsx-tui/env` (0400, holds RSX_GW_JWT_SECRET), exec rsx-tui with
  RSX_TUI_USER + RSX_GW_URL. Secret never in a key line.
- `scripts/rsx-tui-authorize`: add/list/remove key lines (ssh-copy-id
  style). `scripts/rsx-tui.authorized_keys.example`. `make tui-ssh-setup`.
- Functionally tested (fake tui binary): valid/invalid/missing user_id,
  add/list/remove all pass; `bash -n` clean.

**Web path — specified, service deferred.** xterm.js + ttyd PTY + nginx
TLS at rsx.krons.cx. Deferred because ttyd runs one command for all
connections — per-browser identity needs an auth front (rsx-auth) + a
spawn-per-session proxy reusing rsx-tui-dispatch. Config skeleton (ttyd
`--once`, nginx WS-upgrade block) in the spec; real service can't be
stood up/verified this pass.

Spec: `specs/2/54-tui-access.md` (partial), indexed. Open founder
