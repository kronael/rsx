## 04:59 ‚Äî maker root cause (the real one) + naming cleanups

The 127.0.0.1 fix helped but wasn't the whole story; #107 (maker running) kept
failing. Instrumented market_maker `main()` to log its exit signal ‚Äî proved the
maker gets **SIGTERM** (not SIGKILL/OOM). (Aside: my reproductions kept
"exit 1, no output" because `pkill -f market_maker.py` matched the bash tool's
own cmdline and killed the shell ‚Äî use the `[m]arket_maker.py` trick.)

Root cause: the maker is a **managed process**, and `stop_all()` (called by the
safety/fault shards' `/api/processes/all/stop` crash-tests) SIGTERMs every
managed process incl. the maker; it's never restored, so the later readiness
shard sees a dead maker. First tried excluding the maker from `stop_all` ‚Äî but
that broke `play_safety` "all-stop clears topology to red" (it asserts the
maker stops too). Reverted. Correct orthogonal fix (user's "minimal, low ifs"):
the **consumer** ensures its precondition ‚Äî readiness `beforeAll` does a
throw-proof idempotent maker-start. Verified **22/22** readiness in isolation.
Also `_maker_running` now checks real liveness (os.kill), not unreaped
returncode.

Naming (user): `wal_integration` ‚Üí `wal` in rsx-matching (builds clean); and
"no SPSC rings = not a tile" ‚Äî matching reworded as a **loop**, not a
degenerate tile. Cast benches: completing the missing one-way/send-breakdown/
WAL sections so the crate page is finalized.
