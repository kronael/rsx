# Asymmetric Durability

Not all data deserves the same treatment. Treating it equally
is expensive and wrong.

## The three tiers

A fill is the only event that changes a user's balance. Losing
a fill means the user's balance is permanently wrong: a court-
able offense, not a UX inconvenience. So every fill is appended
to the WAL the instant the matching engine produces it, and the
WAL is fsync'd on a 10 ms flush cycle. The fill is forwarded
downstream as soon as it is buffered, not held until the fsync —
so a crash can lose the fills from the last unflushed window.
WAL replay reconstructs everything up to the last fsync on
restart. Loss window: bounded by the 10 ms flush interval.

An order intent is not sacred. It is the client's request, not
the exchange's commitment. If the gateway crashes after receiving
an order but before forwarding it to risk, the order simply
ceases to exist. The client's WebSocket closes. The client
retries. Nothing is permanently wrong because no execution
happened. This is why order intents are not WAL'd — adding
durability to the order path would add latency and persistence
overhead to every trade, for a failure mode the client already
handles by retrying.

Positions are derived state. A user's position is the sum of
their fills since inception. As long as fills are durable,
positions are recoverable by replay, regardless of what happens
to the in-memory state in the risk engine. The risk engine
writes positions to Postgres asynchronously every 10 ms. On
crash and restart, it loads the last Postgres snapshot and
replays fills from the WAL from that point forward. Maximum
data loss on a single process crash: 10 ms of position updates.
Maximum on dual failure (risk and Postgres): 100 ms, bounded
by the Postgres sync interval.

## Why this framing matters

The distinction that does the work is source of truth versus
derived. Fills are source of truth — they are
generated by the matching engine, persisted with sequence
numbers, and cannot be reconstructed from anything else.
Positions are computed from fills, so they are derived. Derived
state can always be recomputed; it does not need its own
independent durability guarantee.

The matching engine's dedup key is `(user_id, order_id)` — the
exchange order id, not the client string. Each acceptance is
persisted in the WAL as `RECORD_ORDER_ACCEPTED`, and an in-memory
window (300 s) rejects a repeat of the same `(user_id, order_id)`
without re-executing; replaying that record rebuilds the dedup set
after a crash. The client-facing idempotency key is the
20-character `cid`, tracked at the gateway, which holds pending
orders by `cid`. So a client can safely retry an order after a
connection drop without risking double execution, even if the
first submission may have executed.

## What the alternative costs

Treating every message as durable — order intents to a queue,
fills to a queue, positions to a database on every update —
is how you build a 20 ms pipeline. You pay four network hops
and four fsyncs per trade. The system is always-durably-
consistent and always-slow.

The asymmetric model pays one fsync per fill (the WAL flush,
amortized across all fills in the 10 ms window) and zero
fsyncs per order intent. The fill path is honest about its
cost — fsync does take time, which is why the matching engine
applies backpressure when the disk falls behind rather than
silently buffering. The order path is honest about its
fragility: clients must handle at-most-once delivery of their
own order submissions.
